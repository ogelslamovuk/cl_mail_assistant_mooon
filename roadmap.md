ROADMAP SUPPORT MAIL

1. Сначала фиксируем baseline проекта

Утверждаем манифест, pipeline и архитектурные принципы проекта.
На этом этапе определяется, какие шаги идут в pipeline, что делается параллельно, что последовательно, и какие сущности являются базовыми: message, thread, case, state.

2. Готовим ТЗ на архитектурный skeleton

Пишем ТЗ не на весь продукт, а только на каркас проекта.
В ТЗ сразу закладываем главное требование: проект должен быть разбит на полуавтономные модули, выделенные отдельными папками согласно согласованному pipeline, плюс отдельный orchestrator для запуска полного контура.

3. Сначала показывается понимание, а не пишется код

До написания кода исполнитель обязан описать:

как он понял задачу;
как будет выглядеть структура проекта;
какие будут папки модулей;
как будет запускаться каждый модуль отдельно;
как будет запускаться весь pipeline через orchestrator.

Только после этого мы проверяем, что структура нас устраивает.

4. После согласования создается skeleton проекта

Создается только каркас:

отдельные папки модулей по pipeline;
общий слой shared/common;
config layer;
state layer;
artifacts/logging;
runners;
fixtures;
отдельный orchestrator.

На этом этапе не нужна полная бизнес-логика.

5. Структура проекта сразу строится по pipeline

Каждый большой этап pipeline должен иметь собственный модуль или группу модулей в отдельной папке.
Смысл в том, чтобы проект был физически разложен по шагам pipeline, а не логически описан на словах в одном большом скрипте.

6. Каждый модуль должен запускаться отдельно

Для каждого модуля должен существовать свой отдельный запуск.
То есть можно проверить только import, только attachment extraction, только enrichment, только retrieval, только decision layer и так далее — без обязательного запуска всего проекта целиком.

7. Общий orchestrator существует отдельно от модулей

Orchestrator не должен подменять собой архитектуру модулей.
Он нужен только для того, чтобы запускать весь pipeline целиком, когда отдельные части уже существуют и умеют работать сами по себе.

8. После skeleton начинаем двигаться строго по одному модулю

Дальше работа идет не по всему проекту сразу, а по одному элементу pipeline за раз.
Берем один модуль, дорабатываем только его, проверяем только его, принимаем только его.

9. Порядок работы с каждым модулем всегда одинаковый

Для каждого нового модуля действует один и тот же цикл:

формулируем ТЗ на конкретный модуль;
исполнитель объясняет понимание;
после согласования пишет код;
модуль запускается отдельно;
модуль проверяется на тестовых и реальных данных;
смотрим artifacts и логи;
если всё ок — фиксируем результат и заливаем правку;
только после этого идем к следующему модулю.

10. Главное правило проекта

Проверили один элемент — зафиксировали — залили правку — пошли дальше.
Нельзя одновременно развивать несколько новых модулей, если предыдущий еще не принят.

11. Первый модуль после skeleton — Mail Import

Сначала проверяем самый первый шаг pipeline: письмо пришло и прошло нормальный intake.

Mail Import должен уметь:

забрать письмо из источника;
сохранить raw_email;
собрать один богатый parsed_email;
вытащить максимум информации из самого письма;
собрать MIME-разбивку;
определить body source;
выставить технические флаги письма;
собрать inventory вложений.

Пока этот слой не работает стабильно и отдельно не проверен, дальше по pipeline не идем.

12. Второй модуль — Attachment Extraction

После этого отдельно поднимаем и тестируем attachment extraction.

Он должен:

работать как отдельный модуль;
на вход получать parsed_email;
обрабатывать attachment inventory;
сохранять допустимые вложения;
определять типы;
извлекать текст / OCR / структуру там, где это возможно;
возвращать inspectable attachment artifacts и extracted content.

13. Третий модуль — Full Identity / Context Enrichment

После этого отдельно поднимаем enrichment.

Он тоже должен работать как отдельный модуль:
на вход получает письмо и результаты attachment extraction,
на выходе возвращает factual context.

14. Четвертый модуль — Case / Thread Binding

После письма, вложений и enrichment отдельно поднимаем case/thread binding.
Он должен связывать новое письмо с существующей историей или создавать новый case/thread.

15. Пятый модуль — Structured LLM Understanding

После этого отдельно поднимаем первый LLM-слой понимания.

На вход LLM должен получать уже полный подготовленный контекст:

message dossier;
extracted attachment content;
factual enrichment;
case / thread context.

Задача этого слоя — не написать ответ, а выдать структурированное понимание кейса.
На выходе должны появляться, как минимум:

summary;
topic;
customer_need;
entities;
confidence;
response_mode.

response_mode является обязательным полем и должен быть одним из:

answer;
ask_clarifying_question;
handoff_to_operator;
no_reply / ignore.

При необходимости сюда же могут входить служебные поля:

risk_level;
needs_human.

16. Шестой модуль — Knowledge Retrieval / RAG

После structured understanding отдельно поднимаем retrieval-слой.

Retrieval запускается всегда после structured understanding как отдельный технический шаг.
Он не включается по решению LLM и не зависит от того, «догадалась» ли модель, что ей нужны дополнительные знания.

Его задача — не принимать решение и не писать ответ, а находить релевантные знания для уже понятого кейса.
На вход он получает structured understanding и полный dossier-контекст.
На выходе возвращает inspectable набор найденных материалов или пустой результат, если релевантного знания не найдено:

регламенты;
правила;
шаблоны;
исторические кейсы;
другие релевантные knowledge fragments.

Важно: retrieval должен быть отдельным техническим слоем, а не неявной магией внутри большого промпта.

17. Седьмой модуль — LLM Decision Layer

После retrieval поднимаем основной decision-layer.

На вход он получает:

message dossier;
structured understanding;
retrieved knowledge / RAG context;
factual enrichment;
case / thread context.

Именно здесь принимается прикладное решение по кейсу.
Этот слой должен:

подтвердить или скорректировать response_mode;
понять, можно ли отвечать сразу;
понять, нужно ли уточнение;
понять, нужен ли handoff оператору;
сформировать reasoning summary для оператора.

Это не route-tree и не жесткая маршрутизация по веткам.
Это LLM-centric decision layer, работающий на полном контексте и ограниченный явными режимами ответа.

18. Восьмой модуль — Draft Builder

После decision-layer отдельно поднимаем draft builder.

Он должен собирать черновик ответа не по сырому письму, а по:

structured understanding;
response_mode;
retrieved knowledge;
decision-layer result.

Черновик может собираться через толстый промпт, шаблонную подачу или их сочетание, но всегда должен опираться на уже собранные факты и найденные правила.

19. Далее модули подключаются по согласованному pipeline

После mail import, attachment extraction, enrichment, case/thread binding, structured understanding, retrieval, decision-layer и draft builder последовательно подключаются остальные шаги:

Telegram operator card;
operator actions;
reply sending;
audit / metrics.

Каждый шаг — отдельный модуль, отдельная проверка, отдельная фиксация.

20. Интеграция идет постепенно

После того как модуль проверен отдельно, он подключается в общий pipeline через orchestrator.
То есть сначала модуль живет автономно, потом становится частью общей цепочки.

21. Тестирование всегда идет в двух режимах

Сначала проверяем модуль изолированно.
Потом проверяем его уже в связке с предыдущими принятыми модулями через orchestrator.

22. Production не собирается одним махом

В production ничего не включается сразу целиком.
Сначала оживает mail import, потом attachment extraction, потом enrichment, потом следующий слой, и так далее, по мере приемки модулей.

23. Финальный режим проекта

В итоге получаем две вещи одновременно:

набор отдельно запускаемых и отдельно проверяемых pipeline-модулей;
отдельный orchestrator, который умеет запускать весь pipeline целиком.

24. Итоговая формула roadmap

Сначала согласуем pipeline и архитектуру.
Потом делаем skeleton с разложением проекта на отдельные папки-модули по pipeline.
Потом каждый модуль разрабатываем, запускаем, тестируем и принимаем отдельно.
После приемки каждого модуля фиксируем результат и заливаем правку.
Общий orchestrator подключает уже принятые модули в единый pipeline.
Дальше двигаемся только шаг за шагом, без big bang.
