МАНИФЕСТ SUPPORT MAIL

1. Что это за проект

Support Mail — это интеллектуальная система первой линии поддержки для входящих клиентских e-mail обращений.

Это:

email-native
modular
stateful
operator-assisted

Это не omnichannel-платформа и не просто генератор ответов.

2. Каналы системы

Клиентский канал:

только e-mail

Внутренний операторский канал:

Telegram как operator control plane

Telegram не является клиентским каналом и используется только для внутренней работы оператора с кейсами.

3. Главная идея системы

Система должна работать не вокруг “написания текста”, а вокруг правильного способа решения обращения.

Главная единица обработки:

не отдельное письмо,
а case / thread со state, историей и логикой обработки.

Даже если на первом этапе мы работаем как будто с автономными письмами, архитектура сразу должна быть thread-aware и case-aware.

4. Базовый pipeline

Пришло письмо и сохранилось как raw.
Mail Import сразу строит нормализованный parsed_email-объект и вытягивает максимум технической информации из самого письма.
После этого Attachment Extraction обрабатывает вложения: сохраняет допустимые файлы, определяет типы, извлекает текст/структуру/OCR там, где это возможно.
Параллельно или следующим ранним слоем Identity / Context Enrichment тянет весь доступный factual context: клиент, заказы, билеты, возвраты, баллы, карта, история кейсов и другие внешние данные.
После этого письмо привязывается к case/thread.
Затем Structured LLM Understanding делает summary, topic, customer_need, entities, confidence и preliminary response_mode уже на базе полного контекста: письмо + extracted attachments + external enrichment + история кейса.
После этого всегда запускается Knowledge Retrieval / RAG как отдельный технический слой поиска релевантных знаний по уже понятому кейсу.
Затем LLM Decision Layer на полном контексте принимает прикладное решение: можно ли отвечать сразу, нужно ли уточнение, нужен ли handoff оператору, и какой response_mode должен быть финальным.
На этой базе Draft Builder собирает черновик.
Telegram показывает оператору карточку кейса.
Оператор подтверждает или меняет решение.
Ответ отправляется клиенту.
Сквозная аналитика и аудит пишутся на всех этапах.

5. Важная архитектурная фиксация

Data lookup не является отдельным route.

Он встроен в ранний слой identity/context enrichment.

То есть:

факты собираются до LLM decision layer;
система не должна принимать важные решения “вслепую”;
retrieval не является скрытой магией внутри большого промпта, а живёт отдельным техническим слоем;
главное прикладное решение по кейсу принимает LLM Decision Layer уже поверх factual context и retrieved evidence.

6. Ранняя обработка письма

Mail Import отвечает не только за получение письма, но и за его безопасный intake / normalization.

На раннем этапе система должна вытащить максимум информации из самого письма без внешних lookup’ов:

Message-ID
In-Reply-To
References
subject
from
to
date
raw headers
body_text
body source selection
наличие text/plain и text/html
MIME parts breakdown
attachment inventory
технические признаки письма

Технические признаки письма могут включать:

is_bounce
is_auto_reply
is_mailing_like
is_system_generated_likely

Для каждого такого признака должны сохраняться explainable reasons.

7. Обработка вложений

Вложения являются first-class input, а не второстепенным приложением к письму.

Система должна как можно раньше:

увидеть вложения;
собрать их inventory;
сохранить допустимые вложения;
распознать тип;
извлечь из них максимум полезного содержимого.

Нужно различать:

attachment inventory / intake
attachment extraction / OCR / parsing
semantic understanding attachment content

Не вся работа с вложениями требует LLM.
LLM подключается после того, как из вложений уже извлечено максимум доступного технического и текстового содержания.

8. Как мы относимся к цепочкам

Система должна воспринимать новое письмо не как изолированный объект, а как новый ход внутри диалога.

При этом этапность разработки такая:

сначала можно стабилизировать pipeline на single-turn / single-message сценариях;
но архитектура сразу должна содержать сущности:
message
thread
case

И должна сразу сохранять:

Message-ID
In-Reply-To
References
subject
from
date
наши исходящие ответы как часть истории кейса

То есть:
цепочки можно внедрять вторым темпом, но место под них должно быть заложено с первого дня.

9. Принцип реализации

Проект не должен быть одним монолитным скриптом.

Он должен строиться как набор независимых модулей, у каждого из которых есть:

своя зона ответственности;
понятный вход;
понятный выход;
отдельный runner;
inspectable artifacts;
возможность изолированного тестирования.

Примеры модулей:

mail import
attachment extraction
identity / context enrichment
case binding
structured llm understanding
knowledge retrieval / RAG
llm decision layer
draft builder
Telegram operator delivery
operator actions
reply sender
audit / metrics

10. Принцип MVP

На MVP используем простые и inspectable носители:

конфиг / классификатор — Excel
runtime / state — Excel
raw / debug artifacts — файлы
knowledge base — Notion

Дальше возможна эволюция:

Excel → Google Sheets
затем → MySQL / постоянное хранилище

11. Принцип разработки

Никакого big bang.

Нельзя пытаться собрать “всю систему сразу”.

Правильный порядок:

сначала архитектурный каркас;
потом модульный skeleton;
потом внедрение по одному слою;
после каждого слоя — отдельная проверка;
следующий этап только после стабилизации предыдущего.

12. Принцип качества

Каждый модуль считается готовым только если:

он делает одну понятную задачу;
у него есть чёткий input / output;
его можно запустить отдельно;
его результат можно глазами проверить;
он не смешивает свою ответственность с соседними слоями.

13. Роль оператора

Оператор — не аварийный костыль, а штатный участник системы.

Он должен:

видеть карточку кейса;
понимать, что распознано;
видеть собранные факты;
видеть финальный response_mode;
видеть reasoning summary;
видеть draft;
иметь право изменить решение системы.

14. Главная формула проекта

Support Mail — это не генератор писем, а модульная case-driven система принятия решений по e-mail обращениям, где до главного decision-layer собираются:
нормализованное письмо,
распознанные вложения,
внешний factual enrichment,
история кейса,
structured understanding,
retrieved knowledge,
а оператор управляет финальным решением через inspectable pipeline.
